## 1. Objective and non-goals

**Objective.** Produce physics-lint rule scores for three locally-available
`laplace-uq-bench` / `diffusion-physics` checkpoints (UNet, FNO, DDPM) on the
first 300 samples of `data/test_in.npz`. Compare those scores against the
three corresponding upstream metric columns (`pde_residual`, `bc_err`,
`max_viol`) from the diffusion-physics README table pinned at commit
`4c2113a`. Ship the result as v1.0's Criterion 3 evidence. One engineering
day, timeboxed.

**Non-goals.**

- Computing a true H⁻¹ residual on Dirichlet-BC Laplace. On non-periodic
  grids physics-lint's `PH-RES-001` falls back to L² trapezoidal by design
  (see `src/physics_lint/rules/ph_res_001.py:123-131`); reaching H⁻¹ would
  require either periodicity (absent here) or a new FE route through
  `MeshField` (v1.1).
- Writing any new physics-lint rule or modifying any existing rule. The
  dogfood uses three existing rules through their public `.check()`
  interfaces unchanged.
- Modifying upstream `diffusion-physics` code. The integration path is
  read-only from diffphys's perspective: we import its training / inference
  helpers or invoke them via subprocess, but we do not add scripts, flags,
  or modules to that repo.
- Re-framing Criterion 3 as "all three axes agree." See §7 for the PASS
  rule — one real-axis agreement on top of the sanity axis is sufficient
  for scoped PASS. The real axes are designed to admit informative
  disagreement.

## 3. Integration architecture

### 3.1 Primary path: two-venv subprocess

**Rationale for primary.** `diffusion-physics` was built against a different
generation of torch / numpy pins than physics-lint. `pip install -e $DIFFPHYS_ROOT`
into physics-lint's `.venv` has a real risk of breaking physics-lint's own
test suite (numpy ABI, torch major version, pydantic / scipy edges). Debugging
a venv-conflict failure after writing integration code against a single-venv
assumption is a larger time sink than setting up two venvs preemptively.
Two-venv is primary; single-venv is a simplification path (§3.2) only if
the conflict turns out to be absent.

**Layout.**

```
$DIFFPHYS_ROOT                 # /Users/zenith/Desktop/diffusion-physics (default)
└── .venv-diffphys/            # NEW: diffphys-only venv; editable-install of diffphys
    └── bin/python

/Users/zenith/Desktop/physics-lint  (this repo)
├── .venv/                     # existing physics-lint venv; UNCHANGED
├── dogfood/
│   ├── run_dogfood_real.py    # NEW: harness, runs in physics-lint's .venv
│   ├── _extract_predictions.py  # NEW: runs in .venv-diffphys via subprocess
│   └── dogfood_real_results.md  # NEW: report artifact
└── ...
```

**Environment variables.**

| Name | Purpose | Default |
| --- | --- | --- |
| `DIFFPHYS_ROOT` | Path to diffusion-physics working tree | `~/Desktop/diffusion-physics` |
| `DIFFPHYS_PYTHON` | Python executable in the diffphys venv | `${DIFFPHYS_ROOT}/.venv-diffphys/bin/python` |

Both are read by `run_dogfood_real.py` at startup; defaults resolve via
`os.environ.get(...)`. Document both in the script's module docstring.

**Interaction.**

1. `run_dogfood_real.py` (physics-lint venv) determines which model to process,
   then invokes `_extract_predictions.py` via `subprocess.run([DIFFPHYS_PYTHON, "dogfood/_extract_predictions.py", "--model-name", ..., "--output", "/tmp/..."])`.
2. `_extract_predictions.py` (diffphys venv) loads the checkpoint via
   `diffphys.model.trainer.{load_config, build_model, _build_ddpm}` and runs
   inference on the first 300 samples of `test_in.npz`. Output: a `.npz`
   file with keys `predictions: (300, 64, 64) float32` and `truth: (300, 64, 64) float32`.
3. `run_dogfood_real.py` loads the `.npz`, applies the three rules per
   problem, aggregates means, checks rankings.

**Subprocess contract.** Fully file-based — no shared Python state between
the two venvs. The subprocess script accepts a small argparse surface
(`--model-name`, `--config`, `--checkpoint`, `--test-npz`, `--n-samples`,
`--max-samples`, `--output`) and returns exit code 0 on success, non-zero
with a stderr message on failure.

**Setup cost estimate.** 45–60 minutes one-time: create `.venv-diffphys`,
`pip install -e $DIFFPHYS_ROOT` into it, smoke-test by running
`_extract_predictions.py --model-name unet_regressor ...` and verifying
the output `.npz` shape and dtype. This lives in preflight P5.

### 3.2 Fallback path: single-venv direct import

If the two-venv setup fails for an unforeseen reason (e.g., diffphys requires
CUDA at import time and the CPU-only env raises), fall back to:

```
pip install -e $DIFFPHYS_ROOT --no-deps  # in physics-lint's .venv
```

If that imports cleanly and `pytest tests/` still passes on physics-lint,
we can import diffphys directly and skip the subprocess layer. The harness
checks the environment at H0 and picks the path automatically; the rest of
the plan (rule application, ranking, report) is path-agnostic.

If both paths fail → Escape E2 (single-venv fallback below the two-venv
primary).

## 4. Preflight gates

**All five must pass before H0 of the 1-day timer.** Failure of any gate
resolves before starting the timer, or escapes to Path B (full deferral).

### P1. DDPM config reproduction (tolerance ±5%)

Run upstream `scripts/evaluate.py` with each candidate config against
`experiments/ddpm/best.pt` and check whether the reported
`test_in.pde_residual.mean` reproduces the README's 4.22:

```
cd $DIFFPHYS_ROOT && $DIFFPHYS_PYTHON scripts/evaluate.py \
    --config configs/ddpm_phase2.yaml \
    --checkpoint experiments/ddpm/best.pt \
    --device cpu
```

Try in order: `configs/ddpm_phase2.yaml`, `configs/ddpm.yaml`,
`configs/ddpm_improved.yaml`. Record which one reproduces and use that
same config throughout Week 2½.

**Tolerance ±5% (±0.21 around 4.22 → accept 4.01–4.43).** Defence: the
published per-problem std is 0.347 (`experiments/ddpm_phase1_results.json`);
the standard error of a 300-problem mean with that per-problem std is
0.347/√300 ≈ 0.020. A different RNG seed on 5-sample DDPM generation adds
a small additional contribution, conservatively bounded by 0.05 total SE.
±5% = ±0.21 ≈ 4× the conservative SE, which is tight enough to reject a
wrong-model reproduction (the next-closest documented DDPM number in the
repo is 4.68 on OOD, well outside the band) while loose enough to accept
stochastic variation from a correct reproduction. ±2% would be inside the
2-sigma noise band for a correct model and would produce spurious failures.

**If none of the three configs reproduce within ±5%** → Escape E1. DDPM
drops from Week 2½. Resulting n=2 is below the threshold for a meaningful
A1 check — fall to Path B (full Criterion 3 deferral).

### P3. Rule signatures (already verified — documented here for implementation reference)

All three rules expose `.check(field: Field, spec: DomainSpec, *, ...) -> RuleResult`.
Signatures are **not** uniform:

- `PH-RES-001.check(field, spec) -> RuleResult` — no kwargs (`ph_res_001.py:53`)
- `PH-BC-001.check(field, spec, *, boundary_target: np.ndarray, abs_threshold=1e-10, abs_tol_fail=1e-3) -> RuleResult` — keyword-only `boundary_target` required (`ph_bc_001.py:33-40`)
- `PH-POS-002.check(field, spec, *, boundary_values: np.ndarray) -> RuleResult` — keyword-only `boundary_values` required (`ph_pos_002.py:23-28`)

Rule-call branching (§6.4) passes the right kwargs per rule. Nothing lives
on the spec.

### P4. README pin and floor availability

  - `rule=PH-RES-001, pde=laplace, grid_shape=(64,64), method=fd4, norm=L2`

  `_load_floor` raises on missing entries and the rule returns before

- `PH-POS-002` loads no floor — binary pass/fail against `1e-10`.

## 6. Axis definitions and per-rule implementation

### 6.1 Domain spec (constructed once, reused across all rule calls)

```python
from physics_lint.spec import (
    DomainSpec, GridDomain, BCSpec, FieldSourceSpec, SymmetrySpec,
)

spec = DomainSpec(
    pde="laplace",
    grid_shape=(64, 64),
    domain=GridDomain(x=(0.0, 1.0), y=(0.0, 1.0)),
    periodic=False,
    boundary_condition=BCSpec(kind="dirichlet"),   # not "dirichlet_homogeneous" — BCs span ±4
    symmetries=SymmetrySpec(declared=[]),
    field=FieldSourceSpec(type="grid", backend="fd", dump_path="unused"),
)
```

Notes on the spec:

- `dirichlet` (not `dirichlet_homogeneous`): verified BC range is [-4.27, 4.28],
  so BCs are non-homogeneous. This also means `preserves_sign=False`, which
  is why `PH-POS-001` SKIPs on this dataset (survey §4).
- `FieldSourceSpec(dump_path="unused")` is a throwaway to satisfy the
  `exactly_one_source` validator. The rules in use do not read
  `spec.field.*` at runtime (confirmed by grep across the three rule sources);
  the field is plumbed explicitly to each `.check()`.

### 6.2 Field construction (per problem)

```python
from physics_lint.field import GridField

pred_field  = GridField(predictions[i], h=(1/63, 1/63), periodic=False, backend="fd")
truth_field = GridField(truth[i],       h=(1/63, 1/63), periodic=False, backend="fd")
```

- `h=(1/63, 1/63)`: matches upstream's `h=1.0/63` pin (`metrics.py:17`).
  Any other `h` breaks PH-RES-001 / `pde_residual` quadrature comparability.
- `periodic=False, backend="fd"`: routes PH-RES-001 into the L² trapezoidal
  branch (`ph_res_001.py:129-131`) and away from the H⁻¹ spectral branch.
- `predictions[i]` is shape `(64, 64)` float32 — matches `values()` contract.

### 6.3 Boundary extraction

Canonical pattern (no manual concatenation, no orientation-convention risk):

```python
boundary_target  = truth_field.values_on_boundary()    # for PH-BC-001
pred_boundary    = pred_field.values_on_boundary()     # for PH-POS-002
```

Both arrays have shape `(252,)` (`Nx + Nx + Ny-2 + Ny-2` for Nx=Ny=64) in
physics-lint's canonical `[left, right, bottom, top]` concatenation order
(`grid.py:152-158`). Because both sides come from `values_on_boundary()`
calls on GridFields with the same shape, ordering matches by construction.

**Orientation note.** Upstream uses image-convention axes (row-0 = top,
col-0 = left); physics-lint's `values_on_boundary()` uses axis-0-as-x
(so physics-lint's "left edge" = upstream's "top row"). This label
mismatch is harmless because:

1. For `PH-BC-001`: we compare `field.values_on_boundary()` (prediction)
   to `boundary_target` (truth, also from `values_on_boundary()`); both
   use the same ordering.
2. For `PH-POS-002`: the rule only reads `boundary_values.min()` and
   `boundary_values.max()` — orientation-agnostic.

We do not use the explicit `bc_top/bc_bottom/bc_left/bc_right` arrays
from `test_in.npz` at all: their values already live on `fields[i]`'s
edges (verified: `fields[0, 0, :] == bc_top[0]`, etc.).

### 6.4 Per-rule call branching

```python
# Three rules, three call shapes. Dispatch explicitly.

res_result = PH_RES_001.check(pred_field, spec)
bc_result  = PH_BC_001.check(pred_field, spec, boundary_target=boundary_target)
pos_result = PH_POS_002.check(pred_field, spec, boundary_values=pred_boundary)
```

Aggregate `raw_value` across the 300 problems per rule → per-model mean:

```python
physlint_scores[model_name] = {
    "PH-RES-001": float(np.mean([r.raw_value for r in res_results])),
    "PH-BC-001":  float(np.mean([r.raw_value for r in bc_results])),
    "PH-POS-002": float(np.mean([r.raw_value for r in pos_results])),
}
```

**PH-POS-002 boundary handling, resolved.** Pass the full 64×64 prediction
as `field` and the prediction's own edge slice as `boundary_values`. Do
*not* strip the field to interior 62×62 — the rule reads `u.min()` /
`u.max()` over the full field, which for a prediction that roughly
satisfies its own BCs returns a value close to `boundary_values.min()` /
`.max()`, so the `below` / `above` overshoot is dominated by interior
excursions. This is slightly stricter than upstream's interior-only scope
(counts any boundary-slice overshoot if present), but the difference is
a documented methodology footnote, not a bug.

### 6.5 Rule-to-column semantic alignment summary

| Rule (raw_value) | Upstream column | Comparability | Expected rank behaviour |
| --- | --- | --- | --- |
| PH-RES-001 (L² trap. of `-Δu`) | `pde_residual` (L² 5-pt stencil RMS of `Δu`) | Same continuous quantity, different quadrature + stencil order + interior-vs-all-points scope | Rankings should agree up to discretization noise. Disagreement = bug in one discretization. |
| PH-BC-001 (L² relative `‖u−g‖/‖g‖`) | `bc_err` (L¹ absolute MAE) | Same physics (boundary trace error), different norms (L² rel vs L¹ abs) | Rankings likely agree — model separation is 150× (DDPM 0.0014 vs FNO 0.2088), well beyond any L¹/L² flip risk |
| PH-POS-002 (max overshoot magnitude) | `max_viol` (interior violation count) | Same physics (max principle), different reporting (continuous vs integer) | Rankings partially tie at zero: UNet and DDPM both have upstream count 0 → physics-lint raw ≈ 0; FNO has count 0.6% → physics-lint raw > 0. Binary collapse: {FNO violates, others don't}. |

## 7. Criterion 3 verdict rule

```python
sanity_pass     = results["PH-RES-001"]["match"]
real_axes_pass  = sum(1 for k in ("PH-BC-001", "PH-POS-002") if results[k]["match"])

if sanity_pass and real_axes_pass >= 1:
    verdict = "PASS (scoped)"
elif not sanity_pass:
    verdict = "BUG — sanity axis failed; investigate discretization before shipping"
elif real_axes_pass == 0:
    verdict = "FAIL — both real axes disagreed with upstream; defer to v1.1"
else:  # sanity_pass and real_axes_pass == 1
    verdict = "PASS (scoped, MIXED) — one real axis disagreed; document"
```

**Why not "all 3 must agree."** The real axes are cross-semantic
comparisons, not reimplementations. Disagreement on one real axis is
potentially informative — e.g., if PH-POS-002 (magnitude) disagrees with
`max_viol` (count) on a near-tie pair, that's a norm-sensitivity finding,
not a bug. The verdict rule is designed to tolerate one informative
disagreement while still producing a scoped PASS. A MIXED verdict gets
a per-axis interpretation paragraph in `dogfood_real_results.md`.

**Binary-mode ranking check for `max_viol`.** Upstream gives
`{UNet: 0%, DDPM: 0%, FNO: 0.6%}` — two-way tie at zero plus one
positive. The ranking check reduces to set equality:

```python
expected_violators  = {"fno"}
physlint_violators  = {m for m, s in physlint_scores.items()
                       if s["PH-POS-002"] > 1e-10}
axis_match = (physlint_violators == expected_violators)
```

The `1e-10` threshold matches `PH-POS-002`'s internal pass/fail boundary.

## 8. Escape hatches (ordered by probable trigger)

### E1. DDPM config reproduction fails (P1) — **most likely preflight failure**

Trigger: None of `ddpm_phase2.yaml`, `ddpm.yaml`, `ddpm_improved.yaml`
reproduces `pde_residual.mean` within ±5% of 4.22.

Action: DDPM drops from scope. Resulting n=2 is below the meaningful
A1 threshold (ranking 2 models is essentially a single-bit comparison).
Collapse to Path B: Criterion 3 deferred to v1.1; document the DDPM
provenance ambiguity as a secondary finding in the fourth-round
tradeoffs entry.

### E4. Sanity axis fails during the ranking check (H3–H4) — **most likely execution failure**

Trigger: After all three models complete, `PH-RES-001` ranking disagrees
with `pde_residual` ranking. Both measure L² on the same data.

Action: Stop. Do not commit the result. This signals a discretization
bug in physics-lint's `PH-RES-001` (or a subtle issue in our
`GridField` construction, e.g., wrong `h`, wrong `backend`, wrong
`periodic` flag). File as a V1 bug; fix; re-run. Shipping a
"disagreement" result on two discretizations of the same continuous
quantity would be wrong.

### E2. Two-venv setup fails (P5) — **moderate likelihood**

Trigger: `.venv-diffphys` setup fails (e.g., editable install errors,
import-time CUDA requirement, filesystem permissions).

Action: Try single-venv direct install (from the physics-lint repo root,
inside its own `.venv`):

```
source .venv/bin/activate                 # physics-lint's venv
pip install -e $DIFFPHYS_ROOT --no-deps
pytest tests/                             # must still pass
```

If single-venv works → skip the subprocess layer and import diffphys
directly in `run_dogfood_real.py`. If single-venv also breaks the test
suite → collapse to Path B.

### E3. DDPM inference exceeds time budget (H3) — **moderate likelihood**

Trigger: DDPM inference with `n_samples=5, T=200, max_samples=300` on
CPU has not completed by H3:30.

Action: Reduce `max_samples` from 300 to 100. Rerun all three models
for consistency (UNet / FNO cost is trivial; same-subset comparison
preserves validity). Document in the commit: "reduced to 100 due to
CPU inference time; the 30× spread on `bc_err` and 6× spread on
`pde_residual` remain directionally valid with n=100 sample-mean
variance."

Do not mix n=300 for some models with n=100 for others — different
subsamples compromise the ranking comparison.

### E5. Hard-stop at H6 — **low likelihood**

Trigger: H6 hits and the harness is not producing a verdict.

Action: Fall to Path B fully. Do not commit partial work. Write a
paragraph for the fourth-round tradeoffs entry describing what got
stuck and why, to inform v1.1 planning.

### 11.4 `docs/tradeoffs.md` — Takeaways

**Replace** the second-round "Clean-clone verification is a Week 1
checklist item" takeaway with:

> **External-artifact alignment requires layer-by-layer verification, not
> a single check.** Four compounding assumptions failed sequentially
> across Week 2 and Week 2½ planning: checkpoints in git; local `.pt`
> files present; published H¹ ranking existing upstream; physics-lint
> producing H⁻¹ on this dataset. Each assumption seemed verified when the
> layer above it was verified. They weren't. The general principle:
> aligned names ≠ aligned semantics ≠ aligned norms ≠ aligned
> quadratures. Every layer of correspondence between physics-lint and an
> external system needs separate verification, line-by-line on both
> sides, against the actual source code of the quantities being compared.
> Week-1 clean-clone verification would have caught layer 1 only; the
> other three layers require reading both systems' implementations of
> the supposedly-aligned quantities. This takeaway generalizes beyond
> physics-lint: any tool claiming to "reproduce" or "compare against"
> another system's output inherits the full stack of alignment
> requirements, and the plan should treat each layer as a separate
> verification item, not an implied transitive property.

**Keep** the second-round takeaways on "central marketing asset is a
forcing function" and "recursive fallback taxonomy" — both still apply.

**Drop** the "load-bearing wording deserves cross-document consistency"
takeaway from the second-round draft. It's subsumed by the
layer-by-layer verification principle.
