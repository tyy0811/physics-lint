## 2026-04-15: pointwise `MeshField.laplacian()` via SPR-recovered double differentiation

**V1 source:** `docs/tradeoffs.md` 2026-04-15 entry; `src/physics_lint/field/mesh.py` module docstring; `docs/plans/2026-04-28-physics-lint-v1-week-3.md` Task 5.

**V1 state:** `MeshField.laplacian()` raises `NotImplementedError`; users are redirected to `MeshField.laplacian_l2_projected_zero_trace()` (interior O(h²) on smooth fields, boundary DOFs hard-pinned to zero by the Dirichlet-condensed L² projection onto `V_{h,0}`).

**V1.1 target:** implement a true pointwise Laplacian on `MeshField` satisfying the `Field.laplacian()` ABC contract via Superconvergent Patch Recovery (SPR) on the gradient field, then differentiated again. References: Zienkiewicz & Zhu 1992 ("The superconvergent patch recovery and a posteriori error estimates"), Babuška & Strouboulis 2001 §3.

**Acceptance criteria:**

- `MeshField.laplacian()` returns a `Field` whose `values()` approximates pointwise `Δu` to O(h²) on smooth analytical solutions (current PH-CON-004 `sin(πx)sin(πy)` and `x(1-x)y(1-y)` test cases).
- Boundary DOFs return non-zero approximations of true boundary `Δu`: specifically, on `x(1-x)y(1-y)` the boundary corner value approximates ≈ −0.5, not 0.
- PH-CON-004's boundary-element exclusion (`tests/rules/test_ph_con_004.py` and `src/physics_lint/rules/ph_con_004.py`) can be removed without changing test outcomes — i.e. boundary elements stop reflecting an operator artifact and start reflecting true conservation behavior.
- A new test file `tests/test_meshfield_laplacian_pointwise.py` validates both criteria independently of PH-CON-004.

**V1.1 implementation cost estimate:** ~1-2 days. Not in V1 because (a) requires its own design pass on the SPR algorithm choice (patch topology, weighting, gradient recovery technique), (b) blocks no V1 functionality — the V1 zero-trace operator is adequate for PH-CON-004's interior-only consumption, (c) Week 3's `MeshField` surface area was already at the spike-validated boundary and expanding it further would have broken the week's schedule hedge.

**Related V1 design discipline:**

- Adding a new `MeshField` operator in V1.1 must follow the "cite the target operation, not the matrices" discipline from `docs/tradeoffs.md`. The new `.laplacian()` is a pointwise pointwise-approximation operator; the existing `.laplacian_l2_projected_zero_trace()` is an L² projection onto `V_{h,0}`. These two operations share function-space ingredients but compute different objects and must retain distinct names.
- The Week-4 Codex review prompt (Task 8 Step 0) includes a formula-vs-plan audit as item (k); the V1.1 pointwise Laplacian work will re-trigger that audit on the new code path.

## 2026-04-17: metrics-compatibility shim — make the A1 sanity axis a byte-identical comparison

**V1 source:** `docs/tradeoffs.md` 2026-04-17 Phase 3 entry; `preflight/2026-04-17_preflight.log` §Investigation; `src/physics_lint/rules/ph_res_001.py`; `dogfood/laplace-uq-bench/src/diffphys/evaluation/metrics.py::pde_residual_norm`.

**V1 state.** Phase 3 executed on 2026-04-17 produced a mechanical `BUG` verdict because physics-lint's PH-RES-001 (fd4 stencil, full 64×64 grid, L² trapezoidal quadrature) and upstream's `pde_residual_norm` (fd2 5-point stencil, interior 62×62 only, dimensionless RMS) compute *definitionally different* quantities. The design doc §6.5 disclosed this ("not a 1:1 reimplementation") but the 4-way verdict rule still fires `BUG` mechanically on a sanity-axis rank mismatch. At the Phase 3 scope (n=100, n_s=1) the UNet/FNO rank-2/3 swap was 3% in physics-lint vs 19% upstream — plausibly inside sampling variance, but impossible to prove without a re-run at upstream scope (15 h DDPM CPU, blows the 6 h timer). The net result: `BUG` verdict, no discretization bug identified, Criterion 3 deferred to v1.1.

**Root cause.** The sanity axis is supposed to be "structurally degenerate" — a direct comparability check between physics-lint and upstream. Today it isn't degenerate because the two implementations deliberately differ on three independent axes (stencil order, scope, norm). Making the sanity axis actually degenerate requires one side to emit the other's exact quantity, byte-identically.

**V1.1 target.** Add a `metrics_compatibility` mode to PH-RES-001 (and as a pattern, to the other rules that have upstream cousins) that emits an auxiliary `upstream_compatible_raw_value` field alongside the default `raw_value`. On Laplace/Dirichlet non-periodic problems, the compatibility mode:
- Uses the classical 5-point fd2 Laplacian stencil (not fd4), with coefficients exactly matching `diffphys/evaluation/metrics.py:23-27`.
- Restricts the residual to the interior `(H-2) × (W-2)` subgrid, excluding the boundary ring.
- Takes the dimensionless RMS `√(mean(r²))` rather than the L² trapezoidal `√(∫|r|² dA)` (drops the `h²` factor).

Expose this via a new rule kwarg (`upstream_compatibility_mode: bool = False`) or a separate comparator helper that the A1 harness can call without polluting PH-RES-001's own default semantics. The A1 harness's sanity axis then compares `upstream_compatible_raw_value` against upstream's `pde_residual_norm`; the verdict rule fires `BUG` only when the two byte-identical-quantity values genuinely rank differently — at which point a sampling-variance-at-reduced-n explanation is no longer available and the ranking mismatch is a real discretization bug to fix.

**Acceptance criteria:**
- `metrics_compatibility` helper or mode exists on PH-RES-001 (Laplace / Dirichlet / non-periodic path only in v1.1; Poisson / heat / wave compatibility is out of scope).
- On the three Phase 3 checkpoints, the compatibility-mode output matches `diffphys/evaluation/metrics.py::pde_residual_norm(field, h=1/63)` to within `1e-6` (fd2 vs fd2 should be bitwise identical modulo float32 reduction order; allow a tiny tolerance for associativity).
- The A1 harness (`dogfood/run_dogfood_real.py`) switches the sanity axis to the compatibility-mode output. The 4-way verdict rule stays the same.
- Regression test: a unit test checks that on `f(x,y) = x*(1-x)*y*(1-y)` (a Poisson solution, not Laplace, but a clean analytic reference), PH-RES-001 default value and compatibility-mode value differ in the expected direction (default larger by the scope factor), and neither is NaN or dominated by boundary stencil noise.
- Re-run Phase 3 under v1.1 scope (n ≥ 300, n_s ≥ 5 for DDPM) against the 6-surrogate set with the compatibility-mode sanity axis. If verdict is still `BUG`, the remaining gap is a real discretization bug to investigate separately.
- README updated to document that physics-lint ships two residual norms per rule (variationally-correct default, upstream-compatible comparison) and when to use each.

**V1.1 implementation cost estimate:** ~0.5–1 day. The fd2 + interior-only + RMS path is 20 lines of numpy; the rule API extension for an auxiliary value is another half-day including tests. The re-run is compute cost, covered by the 6-surrogate dogfood entry above.

**Related V1 design discipline:**
- The default PH-RES-001 value (fd4, full-grid, L²) is the variationally-correct choice per the Bachmayr / Ernst citations and stays the primary rule output; compatibility mode is an *additional* value for direct upstream comparison, not a replacement.
- The same pattern applies to PH-POS-002 (today's Phase 3 showed a binary-axis mismatch between physics-lint's overshoot-magnitude threshold and upstream's violation-count threshold; a `max_viol_count_compatible_value` emission would resolve that axis too).
- This entry is specifically a v1.1 backlog item, not a v1.0 blocker — the design-doc §6.5 disclaimer correctly anticipated the issue, but the plan's verdict-rule wiring did not soften the `BUG` branch when the scope of disagreement was definitional rather than numerical. Future plans that define a cross-tool comparison should either (a) make both sides emit the comparee's exact quantity, or (b) soften the verdict so a definitional gap reads as `INCOMPARABLE` rather than `BUG`.

## 2026-04-17: CLI `check` — auto-extract per-rule kwargs instead of skipping

**V1 source:** Week 4 Task 2 `src/physics_lint/cli/check.py` (signature-based skip + SKIPPED RuleResult emission); Codex review on feature/v1-week-4 Tasks 0-3 (Priority 1 sub-question (c)); Week 4 plan §Task 2 Step 3 "Note on the TypeError branch" (documented V1 limitation); `docs/tradeoffs.md` 2026-04-17 Phase 3 and Codex-review entries (silent-correctness-failure pattern).

**V1 state.** The CLI iterates the rule registry and invokes each rule as `check_fn(loaded.field, loaded.spec)`. Three rules require additional kwargs: `PH-BC-001` needs `boundary_target`, `PH-POS-002` needs `boundary_values`, `PH-NUM-002` needs `refined_field`. The Week 4 CLI detects these via `inspect.signature` before invocation (so rule-internal `TypeError` propagates instead of being swallowed — see Codex Priority 1 (a)) and emits a `SKIPPED` `RuleResult` with `reason="requires <kwargs> (CLI V1 limitation; V1.1 auto-extracts)"`. The skip is visible in text summary (⊘ glyph), JSON output, and SARIF `toolExecutionNotifications` (never in `run.results`, so the Security tab stays clean). From the CLI, PH-BC-001 / PH-POS-002 / PH-NUM-002 do not run in V1.0 — they skip cleanly rather than fire half-configured.

**Why it's a V1 limitation, not a V1 bug.** Two of the three kwargs (`boundary_target`, `boundary_values`) can be derived from the loaded target automatically: `boundary_target` for a Dirichlet BC can default to `field.values_on_boundary()` (the rule then measures self-consistency, which is degenerate but not wrong); `boundary_values` for PH-POS-002 is the same data by a different name. `refined_field` for PH-NUM-002 requires a *second* grid at 2× resolution, which either has to be loaded separately (new loader contract) or computed via re-running a user-supplied model at finer resolution (adapter contract change). The first two are mechanical; the third is a design decision the V1 CLI scope did not budget for.

**V1.1 target.** Extend the CLI dispatch loop to auto-populate `boundary_target` and `boundary_values` from `loaded.field.values_on_boundary()` when the BC is Dirichlet / Dirichlet-homogeneous and the rule's signature asks for them. For PH-NUM-002, either (a) accept a `--refined-target <path>` CLI flag that loads a second dump at finer resolution, (b) extend adapter contract to emit a refined prediction alongside the primary, or (c) skip the rule cleanly with a clearer reason (documented trade-off between "degenerate but informative self-check" and "genuine convergence rate"). Choose (a) or (b) after a cost pass.

**Acceptance criteria:**
- `physics-lint check <target>` invokes PH-BC-001 and PH-POS-002 automatically when the loaded target has a Dirichlet BC; the rules run and produce PASS/WARN/FAIL (not SKIPPED) on a representative laplace-uq-bench dump.
- `--refined-target <path>` CLI flag OR adapter `domain_spec()` emitting a second prediction enables PH-NUM-002 in the automatic dispatch; the rule runs on at least one real surrogate.
- The current signature-based-skip path remains as a fallback for rules whose requirements can't be auto-derived, with the same SKIPPED RuleResult emission the V1 CLI uses.
- Tests: for each auto-extracted rule, a CLI-level test verifying it runs (not SKIPPED) on a dump whose spec satisfies the extraction preconditions.
- No regression on the Codex Priority 1 (a) contract: rule-internal `TypeError` still propagates; the signature-based precheck is the only skip path.

**V1.1 implementation cost estimate:** ~1 day. BC-related extraction is ~1 hour + tests. PH-NUM-002 refined-target design + implementation is ~4–6 hours depending on which option is chosen. Test coverage for auto-extraction path is ~2 hours.

**Related V1 design discipline:**
- The V1 signature-based-skip pattern is not a hack, it is the correct structural fix for the silent-correctness-failure pattern caught in the Codex review (Priority 1 (a)). V1.1 auto-extraction is an *extension* of that pattern — "if we know how to derive the kwarg from the loaded target, we do; otherwise we fall back to SKIPPED." The skip path is retained as the floor, not removed.
- SKIPPED RuleResult emission (Codex Priority 1 (b)) is permanent, not a V1 workaround. Any skipped rule — signature-skipped, spec-mismatched, or explicitly disabled by `--disable` — must always surface in the user-visible summary so users can see which rules did and did not fire.
