## 2026-04-17: metrics-compatibility shim — make the A1 sanity axis a byte-identical comparison

**V1 source:** `docs/tradeoffs.md` 2026-04-17 Phase 3 entry; `preflight/2026-04-17_preflight.log` §Investigation; `src/physics_lint/rules/ph_res_001.py`; `dogfood/laplace-uq-bench/src/diffphys/evaluation/metrics.py::pde_residual_norm`.

**V1 state.** Phase 3 executed on 2026-04-17 initially fired a mechanical `BUG` verdict because physics-lint's PH-RES-001 (fd4 stencil, full 64×64 grid, L² trapezoidal quadrature) and upstream's `pde_residual_norm` (fd2 5-point stencil, interior 62×62 only, dimensionless RMS) compute *definitionally different* quantities. The design doc §6.5 disclosed this ("not a 1:1 reimplementation") but the 4-way verdict rule still fires `BUG` mechanically on a sanity-axis rank mismatch. At the Phase 3 scope (n=100, n_s=1) the UNet/FNO rank-2/3 swap was 3% in physics-lint vs 19% upstream — plausibly inside sampling variance, but impossible to prove without a re-run at upstream scope (15 h DDPM CPU, blows the 6 h timer). After reconsideration (see `docs/tradeoffs.md` 2026-04-17 entry: "Initial E4 reading and recovery" and "Criterion 3 outcome"), the verdict was re-read as **scoped-MIXED PASS** under a softened `INCOMPARABLE` sanity axis rather than `BUG`, and Criterion 3 was met in scoped form. The shim below is the V1.2 path-forward to restore the byte-identical sanity-axis assumption so a future `BUG` reading corresponds to a real discretization bug rather than the pre-disclosed definitional gap.

**Root cause.** The sanity axis is supposed to be "structurally degenerate" — a direct comparability check between physics-lint and upstream. Today it isn't degenerate because the two implementations deliberately differ on three independent axes (stencil order, scope, norm). Making the sanity axis actually degenerate requires one side to emit the other's exact quantity, byte-identically.

**V1.2 target.** Add a `metrics_compatibility` mode to PH-RES-001 (and as a pattern, to the other rules that have upstream cousins) that emits an auxiliary `upstream_compatible_raw_value` field alongside the default `raw_value`. On Laplace/Dirichlet non-periodic problems, the compatibility mode:
- Uses the classical 5-point fd2 Laplacian stencil (not fd4), with coefficients exactly matching `diffphys/evaluation/metrics.py:23-27`.
- Restricts the residual to the interior `(H-2) × (W-2)` subgrid, excluding the boundary ring.
- Takes the dimensionless RMS `√(mean(r²))` rather than the L² trapezoidal `√(∫|r|² dA)` (drops the `h²` factor).

Expose this via a new rule kwarg (`upstream_compatibility_mode: bool = False`) or a separate comparator helper that the A1 harness can call without polluting PH-RES-001's own default semantics. The A1 harness's sanity axis then compares `upstream_compatible_raw_value` against upstream's `pde_residual_norm`; the verdict rule fires `BUG` only when the two byte-identical-quantity values genuinely rank differently — at which point a sampling-variance-at-reduced-n explanation is no longer available and the ranking mismatch is a real discretization bug to fix.

**Acceptance criteria:**
- `metrics_compatibility` helper or mode exists on PH-RES-001 (Laplace / Dirichlet / non-periodic path only in v1.1; Poisson / heat / wave compatibility is out of scope).
- On the three Phase 3 checkpoints, the compatibility-mode output matches `diffphys/evaluation/metrics.py::pde_residual_norm(field, h=1/63)` to within `1e-6` (fd2 vs fd2 should be bitwise identical modulo float32 reduction order; allow a tiny tolerance for associativity).
- The A1 harness (`dogfood/run_dogfood_real.py`) switches the sanity axis to the compatibility-mode output. The 4-way verdict rule stays the same.
- Regression test: a unit test checks that on `f(x,y) = x*(1-x)*y*(1-y)` (a Poisson solution, not Laplace, but a clean analytic reference), PH-RES-001 default value and compatibility-mode value differ in the expected direction (default larger by the scope factor), and neither is NaN or dominated by boundary stencil noise.
- Re-run Phase 3 under v1.2 scope (n ≥ 300, n_s ≥ 5 for DDPM) against the 6-surrogate set with the compatibility-mode sanity axis. If verdict is still `BUG`, the remaining gap is a real discretization bug to investigate separately.
- README updated to document that physics-lint ships two residual norms per rule (variationally-correct default, upstream-compatible comparison) and when to use each.

**V1.2 implementation cost estimate:** ~0.5–1 day. The fd2 + interior-only + RMS path is 20 lines of numpy; the rule API extension for an auxiliary value is another half-day including tests. The re-run is compute cost, covered by the 6-surrogate dogfood entry above.

**Related V1 design discipline:**
- The default PH-RES-001 value (fd4, full-grid, L²) is the variationally-correct choice per the Bachmayr / Ernst citations and stays the primary rule output; compatibility mode is an *additional* value for direct upstream comparison, not a replacement.
- The same pattern applies to PH-POS-002 (today's Phase 3 showed a binary-axis mismatch between physics-lint's overshoot-magnitude threshold and upstream's violation-count threshold; a `max_viol_count_compatible_value` emission would resolve that axis too).
- This entry is specifically a v1.2 backlog item, not a v1.0 blocker — the design-doc §6.5 disclaimer correctly anticipated the issue, but the plan's verdict-rule wiring did not soften the `BUG` branch when the scope of disagreement was definitional rather than numerical. Future plans that define a cross-tool comparison should either (a) make both sides emit the comparee's exact quantity, or (b) soften the verdict so a definitional gap reads as `INCOMPARABLE` rather than `BUG`.

## 2026-04-17: CLI `check` — auto-extract per-rule kwargs instead of skipping

**V1 source:** Week 4 Task 2 `src/physics_lint/cli/check.py` (signature-based skip + SKIPPED RuleResult emission); Codex review on feature/v1-week-4 Tasks 0-3 (Priority 1 sub-question (c)); Week 4 plan §Task 2 Step 3 "Note on the TypeError branch" (documented V1 limitation); `docs/tradeoffs.md` 2026-04-17 Phase 3 and Codex-review entries (silent-correctness-failure pattern).

**V1 state.** The CLI iterates the rule registry and invokes each rule as `check_fn(loaded.field, loaded.spec)`. Three rules require additional kwargs: `PH-BC-001` needs `boundary_target`, `PH-POS-002` needs `boundary_values`, `PH-NUM-002` needs `refined_field`. The Week 4 CLI detects these via `inspect.signature` before invocation (so rule-internal `TypeError` propagates instead of being swallowed — see Codex Priority 1 (a)) and emits a `SKIPPED` `RuleResult` with `reason="requires <kwargs> (CLI V1 limitation; V1.1 auto-extracts)"`. The skip is visible in text summary (⊘ glyph), JSON output, and SARIF `toolExecutionNotifications` (never in `run.results`, so the Security tab stays clean). From the CLI, PH-BC-001 / PH-POS-002 / PH-NUM-002 do not run in V1.0 — they skip cleanly rather than fire half-configured.

**Why it's a V1 limitation, not a V1 bug.** Two of the three kwargs (`boundary_target`, `boundary_values`) can be derived from the loaded target automatically: `boundary_target` for a Dirichlet BC can default to `field.values_on_boundary()` (the rule then measures self-consistency, which is degenerate but not wrong); `boundary_values` for PH-POS-002 is the same data by a different name. `refined_field` for PH-NUM-002 requires a *second* grid at 2× resolution, which either has to be loaded separately (new loader contract) or computed via re-running a user-supplied model at finer resolution (adapter contract change). The first two are mechanical; the third is a design decision the V1 CLI scope did not budget for.

**V1.2 target.** Extend the CLI dispatch loop to auto-populate `boundary_target` and `boundary_values` from `loaded.field.values_on_boundary()` when the BC is Dirichlet / Dirichlet-homogeneous and the rule's signature asks for them. For PH-NUM-002, either (a) accept a `--refined-target <path>` CLI flag that loads a second dump at finer resolution, (b) extend adapter contract to emit a refined prediction alongside the primary, or (c) skip the rule cleanly with a clearer reason (documented trade-off between "degenerate but informative self-check" and "genuine convergence rate"). Choose (a) or (b) after a cost pass.

**Acceptance criteria:**
- `physics-lint check <target>` invokes PH-BC-001 and PH-POS-002 automatically when the loaded target has a Dirichlet BC; the rules run and produce PASS/WARN/FAIL (not SKIPPED) on a representative laplace-uq-bench dump.
- `--refined-target <path>` CLI flag OR adapter `domain_spec()` emitting a second prediction enables PH-NUM-002 in the automatic dispatch; the rule runs on at least one real surrogate.
- The current signature-based-skip path remains as a fallback for rules whose requirements can't be auto-derived, with the same SKIPPED RuleResult emission the V1 CLI uses.
- Tests: for each auto-extracted rule, a CLI-level test verifying it runs (not SKIPPED) on a dump whose spec satisfies the extraction preconditions.
- No regression on the Codex Priority 1 (a) contract: rule-internal `TypeError` still propagates; the signature-based precheck is the only skip path.

**V1.2 implementation cost estimate:** ~1 day. BC-related extraction is ~1 hour + tests. PH-NUM-002 refined-target design + implementation is ~4–6 hours depending on which option is chosen. Test coverage for auto-extraction path is ~2 hours.

**Related V1 design discipline:**
- The V1 signature-based-skip pattern is not a hack, it is the correct structural fix for the silent-correctness-failure pattern caught in the Codex review (Priority 1 (a)). V1.2 auto-extraction is an *extension* of that pattern — "if we know how to derive the kwarg from the loaded target, we do; otherwise we fall back to SKIPPED." The skip path is retained as the floor, not removed.
- SKIPPED RuleResult emission (Codex Priority 1 (b)) is permanent, not a V1 workaround. Any skipped rule — signature-skipped, spec-mismatched, or explicitly disabled by `--disable` — must always surface in the user-visible summary so users can see which rules did and did not fire.
- **V1.0 partial auto-extraction shipped.** The CLI already auto-extracts `boundary_target` and `boundary_values` for (a) dumps that ship a `boundary_target` array in their .npz (`dogfood/make_ci_dumps.py` does this for Week 2½ surrogates) and (b) `dirichlet_homogeneous` BCs (zeros on the boundary). V1.1 extends to non-homogeneous dirichlet without a shipped array, and to `PH-NUM-002` `refined_field`.

## 2026-04-17: CLI per-rule override surface (`[tool.physics-lint.rules]`)

**V1 source:** Codex review on feature/v1-week-4 Tasks 4-6 (Finding 4); `docs/design/2026-04-14-physics-lint-v1.md` §12 CLI + config; Week 4 plan Task 3 `config show` / `config init` subcommands.

**V1 state.** The design doc specifies per-rule override syntax in `[tool.physics-lint.rules]`:

    [tool.physics-lint.rules]
    "PH-RES-001" = { tol_pass = 10.0, tol_fail = 100.0 }
    "PH-BC-001"  = { abs_threshold = 1e-10, abs_tol_fail = 1e-3 }
    "PH-SYM-003" = { enabled = false }
    "PH-CON-003" = { severity = "error" }

`DomainSpec` parses this block (pydantic v2 accepts the nested dict), but the V1.0 CLI does not apply any of the four override kinds — the block is read and ignored. The V1.0 workaround is `--disable <RULE_ID>` at the command line for the `enabled = false` case; `tol_pass`, `abs_threshold`, and `severity` have no V1.0 equivalent.

**V1.2 target.** Wire each override kind through the CLI's rule dispatch:

- `tol_pass` / `tol_fail` / rule-specific floats (`abs_threshold`, `abs_tol_fail`): pass through as kwargs to the rule's `check()` function. Each rule already accepts its thresholds as keyword-only parameters with defaults, so the plumbing is a dict lookup + kwargs merge at dispatch.
- `enabled = false`: treat the rule as if it were in `--disable`.
- `severity = "<level>"`: override `entry.default_severity` on the RegistryEntry before rule invocation, and thread the override through `RuleResult.severity` so SARIF level mapping and `exit_code` computation use the override.

**Acceptance criteria:**

- CLI-level tests for each override kind on at least one real rule (e.g. `PH-RES-001 tol_pass`, `PH-SYM-003 enabled = false`, `PH-CON-003 severity = error`).
- `physics-lint config show` surfaces which overrides were loaded from TOML (already partially done — it prints the full resolved spec — but should add a "overrides active on this run" summary for readability).
- The README `[tool.physics-lint.rules]` documentation can return as a live (not "design-doc future") feature.
- No regression on `--disable RULE_ID`; the CLI flag and the TOML `enabled = false` can coexist (CLI flag wins per override precedence in `merge_into_spec`).

**V1.2 implementation cost estimate:** ~0.5 day. The dispatch plumbing is ~50 LOC; the hardest part is threading `severity` overrides through the report assembly without breaking `exit_code` semantics. Test coverage is another ~0.5 day.

**Related V1 design discipline:**
- The V1.0 README was corrected to mark the `[tool.physics-lint.rules]` block as design-doc future surface, not live v1.0 feature. The same principle applies to any documented config: if the CLI doesn't apply it, the README should not claim it. See the feedback_codex_review memory entry for the pattern.

## 2026-04-17: pre-commit hook multi-target + `--rule` self-test

**V1 source:** Codex review on feature/v1-week-4 Tasks 4-6 (Findings 3 and 5); `.pre-commit-hooks.yaml` V1.0 stub; `src/physics_lint/cli/self_test.py` V1.0 signature.

**V1 state.** Two small CLI UX limitations.

(1) **Pre-commit hook accepts only one target.** The shipped `.pre-commit-hooks.yaml` uses `pass_filenames: false` because `physics-lint check` accepts exactly one target argument and pre-commit by default passes every matched file as an argument — which produces a Typer "unexpected extra argument" error. Downstream users must add `args: [path/to/adapter_or_dump]` in their `.pre-commit-config.yaml` to specify the target. This is a viable pattern (users explicitly choose which adapter/dump the hook lints) but not the "lint all changed files" ergonomic most pre-commit users expect.

(2) **`self-test --rule <id>` not supported.** The V1.0 CLI signature dropped the `--rule` option (previously accepted but ignored) to avoid the misleading-API pattern. The self-test battery runs all rules on all analytical cases or nothing.

**V1.2 target.**

For (1): either extend `physics-lint check` to accept multiple targets and loop internally, OR introduce a new subcommand `physics-lint check-batch <paths...>` used only by the pre-commit hook. The batch path emits one RuleResult set per target and aggregates into a single report (useful for pre-commit output).

For (2): wire `--rule <PH-XXX-NNN>` through `physics_lint.selftest.run()` — filter the analytical cases to those that exercise the named rule and run only those. Useful for fast iteration while debugging a single rule.

**Acceptance criteria:**

- `physics-lint check-batch a.npz b.py c.npz` produces a single aggregate report covering all three targets; pre-commit hook updated to `pass_filenames: true` + `entry: physics-lint check-batch`.
- `physics-lint self-test --rule PH-RES-001` runs only the analytical cases that exercise PH-RES-001 and exits 0 iff those cases pass.
- Existing single-target `physics-lint check` and all-rules `physics-lint self-test` behavior unchanged.

**V1.2 implementation cost estimate:** ~0.5 day each.

## 2026-04-17: PH-BC-001 / PH-RES-001 relative-mode denominator regularization on homogeneous-Dirichlet samples

**V1 source:** Week 4 Task 4 Step 0 interpretation pass (2026-04-17); README §"v1.0 known limitations"; per-model verdict table in commit `955ebf1` message. Artifacts: `dogfood/laplace_uq_bench/{unet_regressor,fno,ddpm}_pred.npz` (sample_index=0, homogeneous Dirichlet).

**V1 state.** `PH-BC-001` in relative mode divides the raw boundary error by `avg|boundary_target|`, with a floor at machine epsilon (~2.2e-16) when the denominator would underflow. On homogeneous-Dirichlet samples (boundary target ≡ 0), the floor dominates, producing `ratio` values of ~1e13–1e14 that trip the FAIL threshold for *any* non-zero raw error. Observed on all three Week 2½ surrogates: UNet raw=1.56e-02 → ratio=7.04e13; FNO raw=1.92e-01 → ratio=8.64e14; DDPM raw=1.04e-02 → ratio=4.70e13. `PH-RES-001` has the same pattern with a 1e-12 floor: UNet raw=45.0 → ratio=4.48e13; FNO raw=51.7 → ratio=5.15e13; DDPM raw=25.6 → ratio=2.55e13.

**Symptoms.** The v1.0 CI workflow on `sample_index=0` of `tyy0811/laplace-uq-bench` shows all three models FAIL `PH-BC-001` and `PH-RES-001`. Raw-value *ordering* is still correct (FNO > UNet > DDPM on both rules, matching upstream `bc_err` and `pde_residual` rankings) and so the Week 2½ 3-axis ranking agreement is unaffected. But absolute `PASS`/`FAIL` verdicts on such samples are threshold artifacts, not physics signals. This is disclosed in the v1.0 README limitations section as a docs entry; the underlying code fix is this v1.2 backlog item.

**V1.2 target.** Regularize the relative-mode denominator with `max(avg|ref|, absolute_floor)` where `absolute_floor` is a calibrated per-problem-class value (e.g. 1e-3 for BC errors on unit-domain Laplace), not machine epsilon. The floor should be loaded from `src/physics_lint/data/floors.toml` alongside the existing rule floors. Consider either:

- **Option A:** add a new `relative_mode_floor` key per rule in `floors.toml`, distinct from the existing `denom_epsilon`. Clearer semantics.
- **Option B:** promote the existing `denom_epsilon` from machine epsilon to a calibrated value. Simpler but loses the distinction between "underflow guard" and "physically-meaningful floor."

Recommend Option A.

**Acceptance criteria:**

- New `floors.toml` key (e.g. `relative_mode_floor`) for `PH-BC-001` and `PH-RES-001`, calibrated via the existing `scripts/calibrate_floors.py` on a representative laplace-uq-bench sample mix.
- On the v1.0 CI dumps (`dogfood/laplace_uq_bench/{unet_regressor,fno,ddpm}_pred.npz`), `PH-BC-001` relative-mode ratios drop from ~1e13–1e14 to ~1e0–1e2 (meaningful magnitudes). Raw-value ordering unchanged.
- On the same dumps, UNet and DDPM `PH-BC-001` move from FAIL → PASS (raw values 0.016 and 0.010 are below threshold when the denominator is a physical floor, not machine epsilon). FNO stays FAIL (raw 0.192 is above threshold). Matches the dogfood ranking narrative.
- `PH-RES-001` likewise: UNet/DDPM move from FAIL → PASS or WARN on `sample_index=0`; FNO stays FAIL.
- New test case `tests/rules/test_ph_bc_001_homogeneous_dirichlet.py` with a synthetic homogeneous-Dirichlet dump that exercises the floor path explicitly and asserts sensible ratio magnitudes.
- v1.0 README limitations section updated to note the v1.2 fix shipped (demoted to historical context, not active limitation).
- CI workflow comments referencing the "all 3 fail" symptom on `sample_index=0` updated.

**V1.2 implementation cost estimate:** ~1 day. Floor calibration is ~2 hours on existing laplace-uq-bench dumps. Rule code changes are ~20 LOC total across `ph_bc_001.py` and `ph_res_001.py`. Test case is ~1 hour. README limitations demotion + CI comment cleanup is ~30 min. Regression check across the full test suite and 3-surrogate dogfood is another ~2 hours.

**Related V1 design discipline:**

- The V1.0 hero pivoted from `PH-BC-001` to `PH-POS-002` during the 2026-04-17 interpretation pass precisely because `PH-BC-001`'s relative-mode FAIL verdict was unreliable on the sample where the screenshot was to be captured. `PH-POS-002` is an absolute-threshold rule and doesn't suffer this issue. V1.1's fix restores `PH-BC-001` to hero-eligible status; the v1.2 README may optionally re-pivot the hero back, depending on which screenshot reads best.
- The floor-calibration pattern in `src/physics_lint/data/floors.toml` was introduced in Week 1/2 for different reasons (avoiding false-positive FAILs on numerically-converged predictions). Extending the pattern to cover the homogeneous-Dirichlet denominator case is a principled use of the same mechanism, not a new hack.
- V1 `docs/tradeoffs.md` should gain a 2026-04-17 entry citing this backlog item as the resolution path for the all-3-FAIL symptom on CI sample 0 — the tradeoff was "ship v1.0 with documented limitation vs hold for fix"; documented limitation won because the rank-ordering is still correct and the hero pivot preserved the marketing narrative.
